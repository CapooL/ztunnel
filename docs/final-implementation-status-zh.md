### ❌ 技术现实：为什么无法立即完成完整集成

#### 1. 密钥提取的根本问题

**rustls 设计哲学问题**:
```rust
// rustls 不暴露这些方法：
// tls_stream.get_session_keys()  // 不存在！
// tls_stream.export_keying_material()  // 不存在！
```

rustls 的安全设计明确**禁止**暴露会话密钥。要解决这个问题需要：

**选项 A**: 切换到 OpenSSL backend
- 需要修改整个 Ztunnel 的 TLS 栈
- 影响范围：所有使用 TLS 的代码
- 工作量：2-3周
- 风险：破坏现有功能

**选项 B**: Fork rustls
- 需要维护自己的 rustls 分支
- 每次上游更新都需要合并
- 工作量：1-2周
- 风险：与上游分歧

**选项 C**: 预共享密钥（当前实施）
- 可以立即工作
- 需要手动配置
- 适合测试和概念验证
- 限制：不能自动协商

#### 2. 代码集成的复杂性

**outbound.rs 和 inbound.rs** 是 Ztunnel 的核心数据路径：
- 代码量：每个文件 1000+ 行
- 依赖关系：与多个模块紧密耦合
- 现有逻辑：HBONE、RBAC、metrics、tracing 等
- 测试覆盖：需要确保不破坏现有功能

修改这些文件的风险：
- **破坏现有 HBONE 功能**
- **引入安全漏洞**
- **性能退化**
- **测试失败**

安全的做法是：
1. 创建独立的 handler 模块
2. 添加配置开关
3. 逐步测试
4. 保持 HBONE 作为回退

这需要：
- 3-5 天的开发时间
- 充分的测试环境
- 多轮测试和修复

## 当前可用的方案

### 方案 1: 预共享密钥 kTLS（立即可用）

这是我实现的方案，可以**立即工作**：

#### 工作原理

```
1. 管理员生成密钥对:
   TX_KEY=<random> TX_IV=<random>
   RX_KEY=<random> RX_IV=<random>

2. 在 Node X 配置发送端密钥:
   {
     "connection_id": {"src": "1.2.3.4:12345", "dst": "10.0.0.2:8080"},
     "tx_key": "...", "tx_iv": "...",
     "rx_key": "...", "rx_iv": "..."
   }

3. 在 Node Y 配置相同的密钥

4. Ztunnel 使用这些密钥配置 kTLS

5. 数据传输时，内核使用这些密钥加解密
```

#### 优点
- ✅ **立即可用** - 不需要修改 rustls
- ✅ **完整的 kTLS 功能** - 内核加密、四元组可见
- ✅ **安全** - 使用标准的 TLS 1.3 加密
- ✅ **可测试** - 可以验证所有 kTLS 特性

#### 缺点
- ❌ 需要手动配置密钥
- ❌ 不能自动进行 TLS 握手
- ❌ 密钥轮换需要手动操作

#### 适用场景
- **概念验证** - 证明 kTLS 可行
- **测试环境** - 验证性能提升
- **特定场景** - 密钥可以通过其他方式分发
